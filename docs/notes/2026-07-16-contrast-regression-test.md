## 背景

PR #34 でコードブロックの配色をパレットへ追従させた際，
シンタックス色のコントラスト担保が手動計算頼みであった．
パレット差し替え時の回帰を `npm test` で検出できるよう，
セルフレビュー指摘を受けて検証テストを別 PR で追加した．

## 検証方式の選定

CSS 変数の解決とコントラスト計算をテキストパースで完結させた．
`node --test` にそのまま載り，ブラウザや PDF ビルドへ依存しない．
検証対象が「変数定義の静的な値」である場合，computed style の
照合（既存の検証手順）より軽量な選択肢になる．

方式の使い分けを表 1 に示す．

表 1．配色検証の方式と適する対象．

| 方式 | 適する対象 | 依存 |
|---|---|---|
| テキストパース＋機械計算 | 変数定義の静的な値どうしの関係 | `node` のみ |
| computed style 照合 | セレクタ適用後の実効値・継承の確認 | ブラウザ環境 |
| 紙面の目視 | レイアウト崩れ・印刷時の見え方 | ユーザー依頼 |

## silent pass を防ぐエラー設計

検査対象が空のまま全件合格と報告する事故を throw で防いだ．
`--code-bg` の欠如と `--palette-code-*` の 0 件をエラーにしている．
トークン改名のリファクタで検査対象が空になっても，テストの失敗で
気づける．「全件合格」と「1 件も検査していない」の区別が要点である．

## var() 解決の対応範囲

`var()` の解決は `var(--x)` の単純参照のみ対応とした．
fallback 付き `var(--x, #fff)` や `calc()` は未対応である．
現行の `palette.css` に出現しない記法への先回り実装は
YAGNI として見送り，必要になった時点でテストから追加する．

## 使用中ブランチへのスタックド PR 手順

Git のブランチは同時に 1 つの worktree でしか使用できない．
親ブランチが別 worktree で使用中の場合，直接コミットできない．
今回は次の手順で回避した．

1. 自分の worktree のブランチ（コミットなし・クリーン）を
   `git reset --hard <親ブランチ>` で親の先端へ合わせる．
2. その上へ作業コミットを積む．
3. `git branch <新名> HEAD` で意味のある名前のブランチを切る．
4. 新名ブランチを push し，親ブランチを base に Draft PR を作る．

親のマージ前に子 PR の base を付け替える運用は
[スタックド PR 運用の落とし穴と復旧手順](2026-07-16-stacked-pr-operations.md)
を参照する．
