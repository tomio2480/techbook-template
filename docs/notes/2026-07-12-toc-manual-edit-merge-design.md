# toc.html 手動編集保持とビルド中断フェイルセーフの設計

Issue #11 対応（PR #15）の設計判断と，レビューで得た知見をまとめる．
`toc.html` の手動編集を保持するマージ方式と，ビルド中断を検知する
フェイルセーフの 2 点が対象である．

## 🔍 背景

`npm run build` は 2 パス構成である．1 パス目で vivliostyle-cli が
`index.html` を生成する．その目次から，2 パス目の直前に
`src/chapters/toc.html` を更新していた．この処理には 2 つの問題が
あった．

- `src/chapters/toc.html` の `<nav>` 内を毎回無条件に丸ごと上書きして
  いた．利用者が手動で追加した項目やリンク文言の言い換えが，ビルドの
  たびに失われていた．
- ビルドが中断されると，1 パス目の英語の自動目次ページを含んだ
  `dist/book.pdf` がそのまま残る．成功したように見えてしまう．

PR #15 でこの 2 点に対応した．[#11](https://github.com/tomio2480/techbook-template/issues/11) をクローズしている．

## 🛠 判断

### toc.html のマージ方式

`updateTocFromIndex()` の全面上書きをやめた．`parseListItems` /
`mergeTocTrees` / `serializeTocTree` によるマージ方式に変更した．

自動生成された木（auto）と既存 `toc.html` の木（old）を比較する．
比較は 2 パスで行う．1 パス目で href とテキストが完全一致するノードを
優先的に確保する．2 パス目で残りに位置ベース・href フォールバックを
適用する．

1 パスのみ（位置ベースのみ）だと問題があった．同一ファイル内（同じ
href を共有する複数見出し）に新規見出しが挿入されただけで，未編集の
既存見出しのテキストが隣の見出しへずれてしまう．誤って上書きされる
高深刻度のバグであり，レビューで発見した．

href を持たないノード同士は，位置以外に同一性を判断する手がかりが
ない．そのため意図的にマッチング対象から除外した．

既存 `<nav>` の `<h1>`〜`<h6>` 見出しも，`extractHeadingOrDefault()`
で保持するようにした．

### ビルド中断フェイルセーフ

`scripts/verify-build.mjs` を新設した．`npm run build` の最終ステップ
で以下 3 点を検証する．

1. ルートに `index.html` が残っていないこと．
2. `dist/book.pdf` の mtime が `dist/.build-marker`（2 パス目直前に
   書き込む marker）以上であること．`<` で失敗判定し，同時刻は許容する．
3. `vivliostyle.config.js` の `entry` が `.md` を参照していること．

### HTML パーサを導入しない判断

新規パーサ依存を追加しなかった．既存の正規表現・文字列の走査方式
（`removeLiContainingHref` と同じ方針）を踏襲した．頑健性は
`node:test` によるユニットテストで担保する方針とした．

## ⚖ 代替案と棄却理由

**toc.html を全面上書きし続け，構造だけ別ファイルに分離する案**
（Issue 本文の対応案 b）は採用しなかった．`toc.html` 自体を手動編集
可能な単一ファイルとして扱う，現行設計の変更コストが大きい．マージ
方式で要件を満たせたため見送った．

**HTML パーサ（cheerio 等）の導入** も見送った．既存実装が正規表現の
みで完結しており，新規依存を追加するコストに見合わないと判断した．
ただし文字列走査ベースの脆さ（`<olfoo>` への前方一致誤マッチ等）は，
レビューで複数回指摘された．タグ名境界チェックの徹底で対応した．

## 💡 詰まった箇所・レビューでの学び

このタスクは gemini-code-assist による自動レビューを 5 ラウンド受けた．
のべ 16 件の指摘のうち 14 件を採用した．詳細は PR #15 の会話を参照．

**子ノードの `<a>` が親ノードに混入するバグ**．「`<a>` タグのない見出
しのテキストが失われる」という指摘（ラウンド 2）への対応中に見つけた．
liHtml 全体に対して `<a>` を正規表現検索していたことが原因だった．
自身に `<a>` を持たない親 `<li>` の中に，子 `<li><a>` がネストする
場合がある．このとき，子の href・テキストを誤って親のものとして
拾ってしまう．指摘されていない別の潜在バグに，対応作業の途中で
気づいた例である．自ノード直下（子 `<ol>` より前）の内容のみを対象
にする設計に修正した．

**CodeQL の「不完全な複数文字サニタイズ」検出**．
`.replace(/<[^>]*>/g, '')` を単発呼び出ししている箇所があった．
`js/incomplete-multi-character-sanitization` として検出された．長さ
8 以下の `{<, >, a}` からなる文字列を全探索したが，この正規表現・
置換パターンで単発置換をすり抜ける入力は見つからなかった．数学的にも
構築が困難であることを確認した．CodeQL のこの指摘は，コードの「形」
に対するヒューリスティックな検出であり，具体的な悪用可能性の証明では
ない可能性が高い．とはいえ CI の必須チェックであり，ループで固定点
まで除去する形は業界標準の対処でもある．そのため素直に修正した．

**同一 href を共有する兄弟ノードの識別限界**．同一チャプターファイル
内の複数見出しは href を共有する．href だけでは同一性を判断できない．
「新規見出し挿入とリネームが同時に発生する複合ケース」は，レビュー
提供の実装を含め，2 パス化後も未解決である．href なしノードには
position 以外に手がかりがないという，構造的な制約による既知の限界
として残っている．対応するには，既存の回帰テストのシナリオを壊さない
設計が別途必要であり，スコープ外として見送った．

**mtime 分解能起因の flaky テスト**．`setTimeout` による実時間待機で
mtime の前後関係を作るテストは，分解能の粗い環境で不安定になりうる．
この指摘を受け，`fs.utimesSync` による明示的な mtime 設定に統一した．
同じ理由で `verifyPdfNewerThanMarker` 自体の比較も，`<=` から `<` に
緩和した．同時刻を許容するようにしている．
