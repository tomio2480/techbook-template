# Issue #14（PDFタグ付与）のサブエージェント委譲で得た知見

Issue #14 対応（[PR #20](https://github.com/tomio2480/techbook-template/pull/20)）
を，実装をサブエージェントへ委譲する形で進めた際に得た知見をまとめる．
特に，モックテストだけでは検出できない実装バグが実機検証で2件見つ
かった経緯を記録する．

## 背景

Issue #14 は，`dist/book.pdf` にアクセシビリティタグを付与する新規
ツール（OpenDataLoader PDF）導入という，規模の大きい作業だった．
`spec-dev`・`code-quality`・`github-dev` の各 Skill に従うよう指示
した上で，実装をサブエージェントへ委譲した．エージェントは
`docs/spec/pdf-tagging.md` の作成，TDD によるスクリプト実装，テスト
57件の作成までを完了し，push 前で作業を止めて報告した．

この時点でテストは全件パスしていたが，CI（`npm run build`）は失敗
した．

## 発見した実装バグ

CI ログを確認したところ，`タグ付き PDF が生成されませんでした` と
いうエラーで失敗していた．原因は次の2点で，いずれもエージェントが
書いたモックベースの単体テストでは検出できないものだった．

### 出力ファイル名の想定違い

実装は `opendataloader-pdf --output-dir <dir> book.pdf` の出力先に
入力と同名の `book.pdf` が書き出されると想定していた．しかし実際に
Java 環境で実行すると，出力ファイル名は `book_tagged.pdf`（拡張子を
除いたベース名に `_tagged` を付与したもの）だった．Issue 本文の PoC
記載にもこの命名規則への言及はなく，公式ドキュメントにも明記がない．

実機で `npx opendataloader-pdf --format tagged-pdf --output-dir <dir>
book.pdf` を直接実行し，生成された一時ディレクトリの中身を確認する
ことで初めて判明した．

### Windows での `npx` 起動失敗

`spawnSync('npx', ...)` は Windows 環境で `ENOENT` となり起動できな
かった．`npx.cmd` に切り替えると今度は `EINVAL` になった．最終的に
は gemini-code-assist のレビュー指摘を受け，`devDependencies` に導
入済みの `node_modules/@opendataloader/pdf/dist/cli.js` を `node` で
直接実行する方式に変更し，`npx`・`shell: true` のいずれも不要にし
た．この方式は起動オーバーヘッドも小さく，Windows 依存の分岐も消え
るため，実装としてもより単純になった．

表 1 に，実機検証で判明した問題と対応を示す．

表 1. 実機検証で判明した問題と対応．

| 問題 | 発見方法 | 対応 |
|---|---|---|
| 出力ファイル名が `book.pdf` でなく `book_tagged.pdf` | CI 失敗ログ＋実機での直接実行 | `resolveTaggedPdfPath` を実際の命名規則に合わせて修正 |
| Windows で `npx` が `ENOENT` | ローカル（Windows）での実行 | `node` による CLI 直接実行へ変更（gemini 指摘が契機） |

## レビュー対応の推移

gemini-code-assist は同一 PR に対して複数ラウンドでレビューする．
本 PR でも次の順で指摘が来た．

1. 初回レビュー（push 直後）: 3件（Windows npx 起動失敗，spawn 失敗
   時のエラーハンドリング，テストの一時ディレクトリ未削除）．
2. 1回目の修正 push 後，数分以内に2回目のレビュー: 1件
   （`npx` から `node` 直接実行への変更提案）．

1回目の指摘に対応した時点で満足せず，push のたびに新規指摘の有無を
確認する運用が実際に効果を発揮した一例である．

## 今後への示唆

- 外部 CLI ツールをラップする実装は，たとえテストが全件パスしてい
  ても，依存ランタイム（今回は Java）込みの実機実行を必ず行う．
  モックは「想定した振る舞い」を検証するだけであり，「実際の振る
  舞いの想定が正しいか」は検証できない．
- サブエージェントへ実装を委譲する場合も，マージ前の実機検証は委譲
  先ではなく委譲元（メインエージョン）の責任として残す．今回は
  push 前の報告を鵜呑みにせず，CI 失敗を契機に実機検証へ踏み込んだ
  ことで2件のバグを本番前に発見できた．
- 新規ツール導入時は，PoC やツールの README に明記されていない挙動
  （出力ファイル名の命名規則，OS 依存の起動方法）が存在しうる前提
  で臨む．
